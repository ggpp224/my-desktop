## 1. 环境准备

### 1.1 必需环境

| 项目 | 要求 |
|------|------|
| Node.js | 18.x 或 20.x（LTS），建议使用 nvm 或官方安装包 |
| Ollama | 已安装并拉取模型，推荐 [Ollama 官网](https://ollama.com) 安装 |
| 系统 | macOS（优先）、Windows（Browser 工具会使用 `start` 打开链接） |

### 1.2 安装 Ollama 与模型

```bash
# 安装 Ollama 后，拉取推荐模型
ollama pull qwen2.5

# 确保服务运行（可选，部分安装方式会自动启动）
ollama serve
# 或交互式运行
ollama run qwen2.5
```

## 2. 安装与启动

### 2.1 安装依赖

在项目根目录（my-desktop）执行：

```bash
npm install
```

会自动执行 `postinstall`，在 `ui/` 下安装前端依赖。

### 2.2 验证 Ollama Tool Calling（可选）

确认本机 Ollama 支持 tool calling 及返回格式：

```bash
npm run verify-ollama
```

需先启动 Ollama。若输出包含 `OK: Ollama returned structured tool_calls` 表示验证通过。

### 2.3 开发模式

一键启动前端、后端与 Electron 窗口：

```bash
npm run dev
```

脚本会先执行 `tsc` 编译，再同时启动：

- Vite 开发服务器（UI）：<http://localhost:5173>
- 后端 API：<http://localhost:3000>
- 待上述服务就绪后启动 Electron，窗口将加载 5173 的页面

若希望分终端分别启动，可执行：

```bash
# 终端 1
npm run dev:ui

# 终端 2
npm run dev:server

# 终端 3（需先执行一次 tsc）
npx tsc
npm run dev:electron
```

### 2.4 生产构建与运行

```bash
npm run build
npm run start
```

- `npm run build`：编译 TypeScript 并构建 UI 到 `ui/dist`。
- `npm run start`：启动 Electron，主进程内会启动 Express 并加载打包后的 UI（不再依赖 5173 端口）。

## 3. 配置说明

所有敏感信息均通过环境变量配置，请勿写入代码或提交到仓库。

| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `OLLAMA_BASE` | Ollama 服务地址 | `http://localhost:11434` |
| `OLLAMA_MODEL` | 使用的模型名 | `qwen2.5` |
| `KB_CHAT_MODEL` | 知识库回答模型（Ollama） | `qwen2.5:7b` |
| `KB_INGEST_MODEL` | 知识库预处理模型（metadata 抽取） | `qwen2.5-coder:14b` |
| `KB_EMBED_MODEL` | 知识库嵌入模型（Ollama） | `bge-m3` |
| `KB_TOP_K` | 最终引用数量上限 | `7` |
| `KB_HYBRID_TOP_K` | 向量/关键词各自候选规模 | `4` |
| `KB_HYBRID_ALPHA` | 混合检索中向量权重（关键词为 1-alpha） | `0.4` |
| `KB_RERANK_MODE` | 重排模式（`rule` 或 `model`） | `rule` |
| `KB_RERANK_MODEL` | 模型重排使用模型名 | `qwen2.5-coder:14b` |
| `KB_RERANK_POOL_SIZE` | 进入重排阶段的候选池大小 | `24` |
| `KB_QUERY_TIMEOUT_MS` | 单次知识库查询超时（毫秒） | `120000` |
| `PORT` | 后端 API 端口 | `3000` |
| `JENKINS_BASE_URL` | Jenkins 根地址（如 https://jenkins.example.com） | 无，不配置则 Jenkins 工具会提示未配置 |
| `JENKINS_TOKEN` | Jenkins API 鉴权 Token（如 Bearer） | 无 |
| `SHELL_CWD` | Shell 工具执行命令时的工作目录 | 当前进程工作目录 |
| `VITE_API_URL` | 前端请求的后端地址（仅前端构建时生效） | `http://localhost:3000` |

示例（开发时可在项目根目录建 `.env` 或导出变量）：

```bash
export JENKINS_BASE_URL=https://jenkins.mycompany.com
export JENKINS_TOKEN=your-token
npm run dev
```

## 4. 界面使用

### 4.1 Chat 面板

- **输入框**：输入自然语言指令，如「开始工作」「打开 Jenkins」「部署 nova」「ws打开base」「关闭cursor的scm」。完整指令见 [可用指令](./可用指令.md)。
- **快捷按钮**：点击「开始工作」「打开 Jenkins」等会直接发送对应文案到 Agent。
- **消息区**：展示用户与 AI 的对话，以及本次调用的工具结果（若有）。
- **流式返回**：聊天默认使用 `/agent/chat/stream`，可实时看到模型增量输出、工具执行进度与 token 统计；异常时自动回退普通请求。
- **知识库问答**：当你输入「某组件怎么用」「文档里如何配置」这类问题时，系统会自动调用本地知识库（默认 `runtime/private-kb/**/*.md`）返回答案，并附带引用片段来源。
- **知识库持久化**：索引会持久化到本地 `runtime/knowledge-index`。服务重启后优先加载本地索引；仅当文档有变更时才重建索引。
- **模型模式切换**：右上角支持「本地 / 外部」切换。本地模式使用 Ollama；外部模式使用 Gemini（可在设置中填写 API Key/模型/Base URL）。
- **主题切换**：右上角 `ThemeSwitcher` 支持多主题切换，选择会写入本地存储，重启后保持。

### 4.1.1 模型设置（Gemini）

- 点击右上角「⚙ 设置」可打开模型设置弹窗。
- 外部模式下可配置：
  - `Gemini API Key`
  - `模型 ID`（默认 `gemini-2.0-flash`）
  - `API 根地址`（可选）
- 点击「连接测试」会调用 `POST /agent/gemini/test` 验证连通性（不落盘）。
- 点击「保存」后仅写入前端本地存储；请求时通过 `body.llm` 传给后端，服务端不写文件。

### 4.2 Workflow 面板

- 列出当前支持的工作流（如 `start-work`）。
- 点击某一工作流名称即可发送「执行工作流 xxx」类指令，由 Agent 调用 workflow 工具执行对应 JSON 配置。
- 对于「开始工作」等场景，支持启动内嵌终端会话：UI 会新增「终端」页签，展示多终端输出并支持实时交互。

### 4.3 Tool 面板

- 展示当前可用的工具名称与简要说明：run_workflow、run_workflow_step、deploy_jenkins、merge_repo、open_in_ide、close_ide_project、open_browser、run_shell。
- **完整指令与示例**见 [可用指令](./可用指令.md)。

### 4.4 Logs 面板

- 显示请求与执行的简要日志（如「发送: …」「Agent 完成」），便于排查问题。

### 4.5 私人知识库页签

- 可通过自然语言「添加私人知识库」或点击相关入口打开「私人知识库」页签。
- 支持选择本地目录并导入其中的 Markdown 文档（调用 `POST /knowledge-base/import`）。
- 导入后可直接触发增量/全量索引重建。
- 在回答引用来源中点击文档，可打开独立文档页签（`KnowledgeDocPanel`），并支持 Markdown 内相对链接跳转。

## 5. 工作流配置

工作流为 JSON 文件，放在项目根目录的 `workflows/` 下，例如 `workflows/start-work.json`：

```json
{
  "steps": [
    { "tool": "shell", "cmd": "docker compose up -d" },
    { "tool": "browser", "url": "https://github.com" },
    { "tool": "browser", "url": "http://localhost:8080" }
  ]
}
```

- **tool**：`shell` | `browser` | `jenkins`。
- **shell** 步骤需带 `cmd`；**browser** 需带 `url`；**jenkins** 需带 `jobName`。
- 执行顺序与 steps 数组一致；任一步失败则中止并返回错误信息。

修改或新增 JSON 后，无需重启应用即可在 Workflow 面板或通过自然语言「执行工作流 xxx」调用（工作流名称为文件名不含 `.json`）。

## 6. 知识库专项说明（重点）

### 6.1 数据来源与导入

- 系统会从 `KB_DOC_DIRS` 指定目录读取 Markdown 文档（默认 `doc,docs,runtime/private-kb`）。
- 通过 `POST /knowledge-base/import` 导入私人知识库后，文档默认写入 `runtime/private-kb/import-latest`。
- 再次导入同一来源时会先清理旧目录，避免旧文件残留造成检索噪音。

### 6.2 重建与增量重建的区别

- **重建知识库索引**：全量重建（所有文档重做预处理 + 向量索引）。
- **增量重建知识库索引**：按文档指纹做增量预处理（未变化文档复用缓存），再构建新索引快照。
- UI 反馈会区分：
  - 历史已缓存
  - 本次已处理
  - 本次复用
  - 当前处理中

### 6.3 查询与排序策略

- 采用 Hybrid Search：向量检索 + 关键词检索融合。
- `KB_HYBRID_ALPHA` 控制融合权重：
  - `0`：全关键词
  - `1`：全向量
  - 默认 `0.4`（关键词权重更高）
- 重排策略：
  - `rule`：规则重排（本地稳定，默认）
  - `model`：本地模型重排（需对应 rerank/chat 能力）

### 6.4 常见问题排查

- **重启后提示文档变化，查询失败**
  - 优先确认 `KB_PERSIST_DIR` 未变更、文档目录未漂移。
  - 执行一次“重建知识库索引”刷新快照。
- **引用来源明显偏题**
  - 先看 `KB_HYBRID_ALPHA` 是否过高偏向向量或关键词。
  - 再看重排模式是否合适（优先 `rule`）。
- **关键规则补充疑似被截断**
  - 开启 `KB_RULE_PATCH_DEBUG=1`，查看候选、入选、missing 判定日志。

## 8. 常见问题

**Q：界面提示「请先安装并启动 Ollama」？**  
A：表示 `/health` 检测到 Ollama 不可用。请确认 Ollama 已安装、已启动（如 `ollama serve`），且已拉取模型（如 `ollama pull qwen2.5`）。

**Q：说「部署 xxx」没反应或报错？**  
A：需配置 `JENKINS_BASE_URL` 和（如需要）`JENKINS_TOKEN`，并确保 Jenkins 可访问。未配置时工具会返回「Jenkins 未配置」类提示。

**Q：Shell 命令在错误目录执行？**  
A：通过环境变量 `SHELL_CWD` 指定工作目录，默认为启动应用时的当前工作目录。

**Q：如何只跑后端 API 做联调？**  
A：执行 `npm run dev:server`（或 `tsx watch server/api.ts`），API 在 3000 端口；前端可单独 `npm run dev:ui` 并设置 `VITE_API_URL=http://localhost:3000`。

**Q：外部模型提示连接失败（Gemini timeout）怎么办？**  
A：优先检查本机网络连通。可在启动后端前设置 `HTTPS_PROXY`（如 `http://127.0.0.1:7890`）并重启；或在设置中填写可访问的 API 根地址。仍失败可调大 `GEMINI_CONNECT_TIMEOUT_MS`。

**Q：「开始工作」里的「打开终端」是打开网页吗？**  
A：不是。会打开本机的 **系统终端（macOS 的 Terminal.app）** 窗口或新页签，并在其中执行命令（如 `cpxy`）。已有终端窗口时会模拟 **Command+T** 新建一个页签再执行。若 macOS 上终端没有弹出或未在新页签执行，请到 **系统设置 → 隐私与安全性 → 自动化** 中允许本应用控制 **Terminal** 和 **System Events**。

**Q：我的工作页签（内嵌终端）关闭后还能恢复吗？**  
A：会话 ID 会暂存到浏览器本地存储。应用聚焦或刷新时会尝试自动恢复；若后端会话已失效会自动清理本地记录。
