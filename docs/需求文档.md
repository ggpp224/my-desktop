# AI Dev Control Center - 需求文档

<!-- AI 生成 By Peng.Guo -->

## 1. 项目概述

| 项目 | 说明 |
|------|------|
| **项目名称** | AI Dev Control Center |
| **目标** | 构建本地运行的桌面应用，通过自然语言或按钮操作，由 AI 自动控制开发者电脑执行日常开发任务。 |
| **运行方式** | 本地 LLM（Ollama）+ Tool Calling，所有操作在本机完成，无云端依赖。 |

## 2. 核心能力

1. **自然语言控制开发流程**：用户输入一句话（如「开始工作」「打开开发环境」），系统理解并自动执行对应操作。
2. **自动执行 Shell 命令**：如启动 docker、启动前端/后端项目等。
3. **自动打开浏览器页面**：如 GitHub、Jenkins、Confluence、Grafana 等。
4. **自动触发 Jenkins 部署**：如「部署 order-service」触发对应 Job。
5. **支持预定义工作流（Workflow）**：通过 JSON 配置多步流程，一键执行。

## 3. 用户场景

### 场景一：开始工作

- **用户输入**：开始工作  
- **系统行为**：
  1. 启动 docker
  2. 启动前端项目
  3. 启动后端服务
  4. 打开 Jenkins
  5. 打开监控页面  

（完整流程可通过工作流 `start-work` 配置；MVP 阶段可先支持单句触发 1～2 步。）

### 场景二：部署服务

- **用户输入**：部署 order-service  
- **系统行为**：调用 Jenkins API 触发对应 Job 的构建/部署。

### 场景三：打开开发环境

- **用户输入**：打开开发环境  
- **系统行为**：在默认浏览器中打开常用工作页面，如 GitHub、Jenkins、Confluence、Grafana（URL 可在配置或工作流中定义）。

## 4. 功能需求

### 4.1 交互方式

- **自然语言**：在 Chat 面板输入文字，由 Agent 解析为工具调用并执行。
- **按钮操作**：提供快捷按钮（如「开始工作」「打开开发环境」），一键发送对应指令或触发固定序列。

### 4.2 工具能力

- **Shell**：在指定工作目录下执行命令，支持安全策略（如限定目录、高危命令确认）。
- **Browser**：按系统（Mac/Windows/Linux）调用系统命令打开 URL。
- **Jenkins**：通过 HTTP API 触发指定 Job 构建，需支持基址与鉴权配置。
- **Workflow**：按名称加载 `workflows/*.json`，顺序执行 shell/browser/jenkins 步骤。

### 4.3 界面模块

- **Chat Panel**：对话输入、AI 回复、执行结果与工具调用展示。
- **Workflow Panel**：工作流列表与「运行」操作。
- **Tool Panel**：当前可用工具说明。
- **Logs Panel**：执行与请求日志，便于排查。

### 4.4 非功能需求

- **安全**：Shell 限定工作目录；敏感配置（如 Jenkins Token）使用环境变量，不硬编码。
- **健壮性**：Ollama 不可用时前端有明确提示；LLM 返回无法解析时降级为提示重试。
- **环境**：支持 Node.js 18+/20+、Ollama（推荐 Qwen2.5）、macOS/Windows。

## 5. 成功标准

- 用户输入「开始工作」后，系统能自动完成：启动开发环境（如 docker）、打开工作页面、准备开发环境（可结合 `start_work.json`）。
- 用户输入「部署 order-service」能触发 Jenkins 部署（需已配置 Jenkins）。
- 用户输入「打开开发环境」能打开 GitHub、Jenkins、Confluence、Grafana 等页面（URL 可在 config 或 workflow 中配置）。

## 6. 约束与边界

- 工作流编辑：当前版本仅支持通过编辑 `workflows/*.json` 文件配置，不提供可视化编辑 UI。
- Shell 高危命令：可要求用户二次确认或通过白名单/黑名单限制，具体策略在实现中约定。
