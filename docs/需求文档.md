# AI Dev Control Center - 需求文档

## 1. 项目概述

| 项目 | 说明 |
|------|------|
| **项目名称** | AI Dev Control Center |
| **目标** | 构建本地运行的桌面应用，通过自然语言或按钮操作，由 AI 自动控制开发者电脑执行日常开发任务。 |
| **运行方式** | 本地 LLM（Ollama）+ Tool Calling，所有操作在本机完成，无云端依赖。 |

## 2. 核心能力

1. **自然语言控制开发流程**：用户输入一句话（如「开始工作」「打开开发环境」），系统理解并自动执行对应操作。
2. **自动执行 Shell 命令**：如启动 docker、启动前端/后端项目等；支持在系统终端（Terminal.app）中执行指定步骤（工作流中 `visible: true`）。
3. **自动打开浏览器页面**：如 GitHub、Jenkins、Confluence、Grafana 等；支持快捷「打开 Jenkins」及直接打开任意 URL。
4. **自动触发 Jenkins 部署**：如「部署 order-service」触发对应 Job；支持预定义 Job 快捷部署（nova、cc-web、react18、biz-solution、biz-guide、scm）及构建参数，部署后支持轮询并展示构建状态（排队中/构建中/成功/失败）。
5. **支持预定义工作流（Workflow）**：通过 JSON 配置多步流程，一键执行；工作流步骤支持 shell（含 `visible` 在终端中执行）、browser、jenkins。
6. **一键合并流程（已实现）**：针对固定项目（nova、biz-solution、scm）的一键合并到目标分支，含拉取、合并、推送及可选 release；步骤通过 SSE 流式输出到前端。

## 3. 用户场景

### 场景一：开始工作

- **用户输入**：开始工作  
- **系统行为**：
  1. 启动 docker
  2. 启动前端项目
  3. 启动后端服务
  4. 打开 Jenkins
  5. 打开监控页面  

（完整流程可通过工作流 `start-work` 配置；MVP 阶段可先支持单句触发 1～2 步。）

### 场景二：部署服务

- **用户输入**：部署 order-service  
- **系统行为**：调用 Jenkins API 触发对应 Job 的构建/部署。

### 场景三：打开开发环境

- **用户输入**：打开开发环境  
- **系统行为**：在默认浏览器中打开常用工作页面，如 GitHub、Jenkins、Confluence、Grafana（URL 可在配置或工作流中定义）。

### 场景四：快捷部署与状态查看

- **用户操作**：在 Chat 面板选择「快捷部署」下拉中的任务（如「部署 nova」「部署 scm」）并触发。  
- **系统行为**：调用 Jenkins API 触发对应 Job（带预定义分支等参数），返回队列/构建信息；前端轮询部署状态并展示「排队中」「构建中… 50%」「部署成功」等。

### 场景五：一键合并

- **用户操作**：点击「合并」菜单中的「合并 nova」「合并 biz-solution」或「合并 scm」。  
- **系统行为**：在对应项目目录执行「当前分支 → 切目标分支 → 拉取 → 合并 → 推送 →（部分项目）pnpm run release」；每步通过 SSE 流式输出到 Logs，冲突时中止并提示。

## 4. 功能需求

### 4.1 交互方式

- **自然语言**：在 Chat 面板输入文字，由 Agent 解析为工具调用并执行。
- **按钮操作**：提供快捷按钮（如「开始工作」「打开开发环境」），一键发送对应指令或触发固定序列。

### 4.2 工具能力

- **Shell**：在指定工作目录下执行命令，支持安全策略（如限定目录、高危命令确认）；另支持「在系统终端中执行」（如 macOS Terminal.app 新窗口/新页签），供工作流中 `visible: true` 的步骤使用。
- **Browser**：按系统（Mac/Windows/Linux）调用系统命令打开 URL；后端提供 `POST /open-url` 供前端直接打开任意 URL。
- **Jenkins**：通过 HTTP API 触发指定 Job 构建，支持基址、Basic/Token 鉴权及 Crumb；支持带构建参数（如 `BRANCH_NAME`）；支持部署后状态查询（队列 URL 或 Job 名 + buildHistory），返回排队中/构建中/成功/失败及进度百分比。
- **Workflow**：按名称加载 `workflows/*.json`，顺序执行步骤；步骤类型为 shell（支持 `visible` 在终端执行）/ browser / jenkins；工作流名支持下划线转连字符（如 `start_work` → `start-work.json`）。
- **合并流程（后端能力）**：针对固定项目（nova、biz-solution、scm）的 Git 合并流程，含切分支、拉取、合并、推送及可选的 `pnpm run release`；通过 SSE 流式返回每步结果，供前端「合并」菜单调用。

### 4.3 界面模块

- **Chat Panel**：对话输入、AI 回复、执行结果与工具调用展示；快捷按钮（如「开始工作」「打开 Jenkins」）；「合并」菜单（合并 nova / biz-solution / scm），流式展示合并步骤；「快捷部署」下拉（预定义 Jenkins 任务）及部署状态轮询展示（排队中/构建中/成功/失败）；支持直接打开 URL。
- **Workflow Panel**：工作流列表与「运行」操作。
- **Tool Panel**：当前可用工具说明（run_shell、open_browser、deploy_jenkins、run_workflow）。
- **Logs Panel**：执行与请求日志（含合并步骤流式输出），便于排查。

### 4.4 非功能需求

- **安全**：Shell 限定工作目录；敏感配置（如 Jenkins 基址、JENKINS_USERNAME、JENKINS_TOKEN）使用环境变量，不硬编码。
- **健壮性**：Ollama 不可用时前端有明确提示；LLM 返回无法解析时降级为提示重试；Jenkins 需登录/无权限时返回明确错误提示。
- **环境**：支持 Node.js 18+/20+、Ollama（推荐 Qwen2.5）、macOS/Windows；macOS 上「在终端中执行」需在系统设置中授权应用控制 Terminal 与 System Events。

## 5. 成功标准

- 用户输入「开始工作」后，系统能自动完成：启动开发环境（如 docker）、在终端中启动前端/后端等（可结合 `start-work.json` 中 `visible: true` 步骤）。
- 用户通过自然语言「部署 xxx」或快捷部署下拉能触发 Jenkins 部署（需已配置 JENKINS_BASE_URL 等）；触发后可轮询并展示部署状态（排队中/构建中/成功/失败）。
- 用户输入「打开开发环境」或点击「打开 Jenkins」等能打开指定 URL（GitHub、Jenkins、Confluence、Grafana 等可在配置或工作流中定义）。
- 用户通过「合并」菜单执行「合并 nova」「合并 biz-solution」或「合并 scm」时，能流式看到每步日志，并在冲突时得到明确提示。

## 6. 约束与边界

- **工作流编辑**：当前版本仅支持通过编辑 `workflows/*.json` 文件配置，不提供可视化编辑 UI。工作流中 shell 步骤可设置 `visible: true` 以在系统终端中执行（便于长时间运行或需交互的场景）。
- **Shell 高危命令**：支持高危命令检测（如 `rm -rf /`），`requireConfirmation` 为占位策略，具体确认交互可在后续实现。
- **合并流程**：当前合并能力针对固定项目与分支（nova→test、biz-solution/scm→test-260127），项目路径与目标分支在代码中配置，非通过环境变量或 UI 配置。
- **Jenkins 预定义**：快捷部署的 Job 列表及构建参数在服务端预定义（如 api 层 JENKINS_JOB_PRESETS），仅 nova 的 Job 名可通过 `JENKINS_JOB_NOVA` 覆盖。
