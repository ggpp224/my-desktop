## 1. 系统架构

### 1.1 总体架构

```
用户
  ↓
Desktop UI (Electron + React)
  ↓
Node 后端 (Express)
  ↓
Agent 模块
  ↓
本地/外部 LLM (Ollama / Gemini)
  ↓
Tool Router
  ↓
Tools (Shell / Browser / Jenkins / Workflow / Knowledge / Jira / Wiki / Cursor)
```

- **Desktop UI**：Electron 承载 React 应用，开发时加载 Vite dev server，生产时加载打包后的静态资源。
- **后端**：Express 提供 HTTP API；生产环境下由 Electron 主进程启动，开发时可单独进程。
- **Agent**：接收用户消息，调用 Ollama/Gemini，解析 `tool_calls`，经 Tool Router 执行工具，将结果回传 LLM 或直接返回前端。
- **Tools**：统一由 Tool Router 按名称分发，各自无状态、可独立测试。
- **UI ViewModel 层**：前端采用 `view` / `viewmodel` / `domain` / `infrastructure` 分层，降低组件与业务逻辑耦合。

### 1.2 数据流

1. 用户在前端输入或点击快捷按钮 → `POST /agent/chat`，body `{ "message": "..." }`。
2. Agent 将 message 与 tools schema 发给 Ollama `/api/chat`。
3. Ollama 返回 `message.tool_calls`（或纯文本）；Agent 解析为 `{ name, arguments }` 列表。
4. Tool Router 根据 `name` 调用对应 tool（shell/browser/jenkins/workflow），传入 `arguments`。
5. 工具执行结果写回 messages，再次调用 Ollama 生成最终回复（或直接返回）。
6. API 返回 `{ success, text?, toolResults?, error? }`，前端展示在 Chat 与 Logs 中。

## 2. 模块设计

### 2.1 Agent 层（agent/）

| 文件 | 职责 |
|------|------|
| `agent.ts` | 对外 `runAgent(message)`：组 messages、调 Ollama、解析 tool_calls、循环执行并回传结果，返回统一结果结构。 |
| `ollama-client.ts` | 封装 `chatWithTools(messages, tools)`、`healthCheck()`、`parseToolCalls(message)`，与 Ollama API 对接。 |
| `gemini-client.ts` | 封装 Gemini 请求与连接测试（外部模型模式）。 |
| `ollama-runtime.ts` | 管理 Ollama 当前激活模型、已安装模型列表与模型切换/卸载。 |
| `tool-router.ts` | `routeAndExecute(call)`：根据 `call.name` 分发到 shell/browser/jenkins/workflow，返回执行结果。 |
| `tools-schema.ts` | 定义传给 Ollama 的 tools JSON schema（name、description、parameters），与各 tool 实现一一对应。 |
| `tool-progress.ts` | 定义工具执行进度事件，供流式接口实时推送。 |

### 2.2 Tools 层（tools/）

| 工具 | 接口 | 说明 |
|------|------|------|
| shell-tool | `run(command, options?)` | `child_process.exec`，cwd 来自 config，支持高危命令检测与二次确认占位。 |
| shell-tool | `runInTerminal(command)` | 在系统终端中执行（macOS Terminal.app 新窗口/新页签，Windows cmd，Linux xterm），供工作流 `visible: true` 步骤使用。 |
| browser-tool | `open(url)` | 按平台使用 `open` / `start` / `xdg-open` 打开 URL。 |
| jenkins-tool | `deploy(jobName, parameters?)` | 触发构建：无参数时 `POST .../build`，有参数时 `buildWithParameters`；支持 Basic/Crumb 鉴权；返回 `queueUrl` 供前端轮询。 |
| jenkins-tool | `getDeployStatus(queueUrl)` | 根据队列 URL 查询排队/构建状态。 |
| jenkins-tool | `getDeployStatusByBuildHistory(jobName)` | 根据 Job 名请求 buildHistory/ajax 解析 HTML，返回最新构建状态及进度百分比。 |
| workflow-tool | `runWorkflow(name)` | 从 `workflows/{name}.json` 读取 steps，顺序执行；shell 步骤支持 `visible` 则调用 `runInTerminal`，否则 `shellRun`；某步失败即中止。 |
| merge-tool | `mergeNova` / `mergeBizSolution` / `mergeScm` | 固定项目的 Git 合并流程（切分支、pull、merge、push、可选 pnpm run release），通过 `onStep` 回调流式输出；由后端 API 直接调用，不通过 Agent。 |

### 2.3 工作流格式（workflows/*.json）

```json
{
  "steps": [
    { "tool": "shell", "cmd": "docker compose up -d" },
    { "tool": "shell", "cmd": "yarn dev", "visible": true },
    { "tool": "browser", "url": "https://example.com" },
    { "tool": "jenkins", "jobName": "my-job" }
  ]
}
```

- **step 类型**：`shell`（需 `cmd`，可选 `visible: true` 在系统终端执行）/ `browser`（需 `url`）/ `jenkins`（需 `jobName`）。顺序执行，单步失败即中止。
- 工作流名称对应文件名（不含 `.json`），Agent 或前端传 `start_work` 时会自动匹配 `start-work.json`。

### 2.5 配置（config/default.ts）

- **ollama**：`baseUrl`、`model`（环境变量 `OLLAMA_BASE`、`OLLAMA_MODEL`）。
- **server**：`port`（`PORT`）。
- **jenkins**：`baseUrl`、`username`、`token`（`JENKINS_BASE_URL`、`JENKINS_USERNAME`、`JENKINS_TOKEN`）；Basic 认证时同时使用 username 与 token；部署预定义由 `config/projects` 中带 `jenkins` 的条目派生。

## 3. 前端设计

### 3.1 技术栈

- React 18 + Vite + TypeScript；构建产物由 Electron 加载（生产时 `loadFile(ui/dist/index.html)`）。

### 3.2 页面结构

- **Header**：标题 + Ollama 未就绪提示（依赖 `GET /health`）+ 主题切换 + 本地/外部模型切换 + Gemini 设置弹窗。
- **左侧边栏**：Workflow Panel（工作流列表与运行）+ Tool Panel（工具列表）。
- **主区域**：Chat Panel + 下方 Logs Panel。  
  **Chat Panel** 包含：快捷按钮（如「开始工作」「打开 Jenkins」）；「合并」菜单（合并 nova / biz-solution / scm），调用 `/merge/*` SSE 并流式写 Logs；「快捷部署」下拉（预定义 Jenkins 任务），调用 `POST /jenkins/deploy` 后轮询 `GET /jenkins/deploy/status` 展示状态；消息列表与输入框；打开 URL 时调用 `POST /open-url`；默认走流式对话接口。
  **多页签**：支持 `workspace`、`my-work`（内嵌终端）、`knowledge-base`（私人知识库）、`knowledge-doc:*`（来源文档）并存。

### 3.3 与后端交互

- 基础 URL：Electron 下通过 `window.electronAPI.getApiBase()` 获取，否则 `VITE_API_URL` 或 `http://localhost:3000`。
- 启动时请求 `GET /health` 决定是否展示 Ollama 提示。
- 发送指令：优先 `POST /agent/chat/stream`，实时消费 `llm_delta`、`tool_progress`、`token_usage`，最终 `result` 结构与 `/agent/chat` 对齐。
- 模型设置：`GET /agent/ollama/models`、`GET/POST /agent/model`、`POST /agent/gemini/test`。
- 指令历史：`GET/POST /agent/history`，持久化最近输入。
- 知识库：`POST /knowledge-base/import` 导入文档，`GET /knowledge-base/document` 打开来源文档全文。
- 合并：`POST /merge/nova|biz-solution|scm`，读取 SSE 流，按 `data: { step }` 写 Logs，`data: { done, success, error }` 结束。
- 部署状态：触发部署后根据返回的 `queueUrl` 或 `jobName` 轮询 `GET /jenkins/deploy/status`，展示排队中/构建中/成功/失败及进度。
- 内嵌终端：`POST /workflow/:workflowName/embedded` 启动会话，后续通过 session/terminal 系列 API 轮询输出与双向交互。

## 4. 安全与错误处理

- **Shell**：仅在配置的 `allowedCwd` 下执行；高危命令可通过 `requireConfirmation` 或后续白名单/黑名单限制。`runInTerminal` 在用户系统终端中执行，依赖系统权限（如 macOS 自动化授权）。
- **敏感信息**：Jenkins 基址、用户名、Token 及 Ollama 等仅通过环境变量或 config 注入，不写入代码与仓库。
- **Ollama 不可用**：`/health` 返回 503 或失败时，前端展示「请先安装并启动 Ollama」及文档链接。
- **LLM 无 tool call**：Agent 将纯文本回复或「未解析到可执行操作」返回前端，不抛未捕获异常。
- **Workflow 单步失败**：当前策略为中止并返回错误信息，后续可扩展「跳过/重试」配置。
- **Jenkins**：请求返回登录页或错误页时，解析为「需登录或无权访问」类提示；状态轮询支持 queueUrl 与 jobName（buildHistory）两种方式。
- **合并流程**：冲突时 `git merge --abort` 并切回原分支，通过 SSE 返回错误信息；前端可对「代码有冲突，需手工合并」做弹窗提示。

## 5. 项目目录与构建

- 根目录即项目根（my-desktop），包含 `agent/`、`tools/`（含 `shell-tool`、`browser-tool`、`jenkins-tool`、`workflow-tool`、`merge-tool`、`workflow-embedded-service`、`terminal-session-service` 等）、`workflows/`、`server/`、`desktop/`、`config/`、`ui/`、`scripts/`、`docs/`。
- 构建：`tsc` 输出到 `dist/`（不含 ui）；`ui` 使用 Vite 构建到 `ui/dist`。Electron 入口为 `dist/desktop/electron-main.js`，生产时主进程在该入口中启动 Express 并加载 `ui/dist`。
